# Proto 기준 개선안 정리 (현황 탭 / 사이클데이터 탭)

## 범위
- 기준 코드: `DataTool_dev/DataTool_optRCD_proto_.py`
- 대상 탭:
  - 현황 탭 (`tb_*` 계열)
  - 사이클데이터 탭 (`indiv/overall/link cycle`, `step/rate/chg/dchg/continue/dcir` 계열)

---

## 1) UX / UI 개선

### 1-1. 현황 탭 검색 UX 고도화
- 현황:
  - 검색은 `Return` 입력 시에만 반영됨 (`FindText.returnPressed.connect`) 
  - 비매칭 셀을 강하게 탈색하여 정보 인지가 급격히 떨어짐
- 근거:
  - `FindText.returnPressed.connect(self.tb_cycler_combobox)` (약 9368행)
  - `tb_channel` 전 셀 재색칠 루프 (약 13317~13368행, 13473~13522행)
- 개선:
  - 디바운스(예: 250~350ms) 기반 실시간 검색
  - `"매칭만 보기"` 토글 추가 (비매칭 숨김/접기)
  - 강조 정책 분리: 배경/글자색/테두리 옵션을 사용자 설정화
- 기대효과:
  - 검색 반복 시간 단축, 대시보드 가독성 증가

### 1-2. 현황 탭 정보 구조 재정렬
- 현황:
  - `tb_summary`, 상단 상태 테이블(`tableWidget`), 대형 채널 그리드(`tb_channel`)가 동시에 고정 렌더됨
  - 정보 밀도는 높은데 우선순위가 명확하지 않음
- 근거:
  - UI 구성부 `tb_summary`, `tableWidget`, `tb_channel` 초기화 (약 3149~3283행)
- 개선:
  - 상단: 핵심 KPI 4개(사용중/대기/완료/이상) 카드화
  - 중앙: 채널 상태 그리드
  - 우측 슬라이드 패널: 선택 채널 상세(testname/path/temp/cycle)
  - 색상 의존도를 낮추고 아이콘/텍스트 뱃지 추가
- 기대효과:
  - 초보자 온보딩 개선, 상태 해석 시간 단축

### 1-3. 사이클데이터 탭의 작업 상태 투명성 강화
- 현황:
  - 진행률은 `progressBar` 단일 값 위주, 어떤 단계(로딩/전처리/플롯/저장)인지 불명확
- 근거:
  - 다수 함수에서 `self.progressBar.setValue(...)` 반복 (예: 10491, 10660, 10831, 11918행 등)
- 개선:
  - 단계형 진행 표시: `데이터 탐색 -> 로딩 -> 계산 -> 플롯 -> 저장`
  - 현재 파일/채널/사이클 카운트 텍스트 동시 표기
  - 취소 버튼 + 안전 중단(현재 future 완료 후 종료)
- 기대효과:
  - 체감 속도 개선, 긴 작업 중 이탈 감소

### 1-4. 탭 생성/그래프 생성 정책 일관화
- 현황:
  - `indiv/overall/link`에서 탭 생성 시점과 유효 데이터 체크 로직이 분산
  - 비유효 데이터 시 figure close 처리 분기 많음
- 근거:
  - `indiv_cyc_confirm_button`, `overall_cyc_confirm_button`, `link_cyc_confirm_button` 내부 분기 (약 10870~11360행)
- 개선:
  - `PlotBuildContext`(데이터 존재/탭/캔버스/toolbar/채널맵) 공통화
  - finalize 정책 단일 함수화(현재 `_finalize_cycle_tab`는 부분 적용)
- 기대효과:
  - UI 버그/누락 탭/메모리 누수 리스크 감소

## 2) 데이터 처리 속도 개선

### 2-1. Toyo DCIR 파일 반복 I/O 캐시
- 현황:
  - `toyo_cycle_data`에서 dcir cycle별로 파일 존재 확인 + `read_csv` 반복
- 근거:
  - `for cycle in cycnum: if os.path.isfile(...): pd.read_csv(...)` (약 837~847행)
- 개선:
  - 채널 단위 LRU 캐시(파일번호 -> 최소 컬럼 DataFrame)
  - 다회 호출 경로(`indiv/overall/link`) 간 공유 캐시 주입
- 기대효과:
  - 디스크 I/O 대폭 감소 (중복 분석 시 큰 효과)

### 2-2. `groupby().apply()` 대체로 벡터화
- 현황:
  - Toyo 전처리에서 `groupby(...).apply(merge_rows)` 사용
- 근거:
  - `Cycleraw.groupby(...).apply(...)` (약 813행)
- 개선:
  - `agg` + 후처리로 치환 (`sum/first/last` 조합)
  - 조건별 컬럼 재계산만 최소한의 numpy/pandas 연산으로 수행
- 기대효과:
  - 중대형 데이터셋에서 CPU 시간 절감

### 2-3. PNE `pivot_table` 다중 호출 통합
- 현황:
  - 유사 인덱스/조건에 대해 `pivot_table` 다회 생성
- 근거:
  - `pne_simul_cycle_data`에서 `max_cap/max_vol/min_vol/min_crate/avg_temp` (약 1827~1835행)
  - `pne_cycle_data`의 `pivot_data` (약 2003행)
- 개선:
  - 동일 집계는 한 번의 groupby/agg 결과에서 파생
  - 자주 쓰는 계산 결과를 구조체로 캐싱
- 기대효과:
  - CPU/메모리 사용량 감소

### 2-4. 공통 subfolder 스캔 캐시 확대
- 현황:
  - 일부 함수는 `subfolder_map` 캐시 사용, 일부는 각 함수별 `os.scandir` 반복
- 근거:
  - `_load_all_cycle_data_parallel`는 캐시 있음 (약 10635~10643행)
  - 다른 프로파일 계열 함수에서 반복 `os.scandir` (예: 10467, 10542, 11900, 12060행 등)
- 개선:
  - 세션 단위 `folder_tree_cache` 도입
  - 경로 변경/리로드 버튼에서만 invalidate
- 기대효과:
  - 대규모 경로에서 탐색 오버헤드 감소

### 2-5. 엑셀 출력 배치화
- 현황:
  - 루프 내부에서 `to_excel`을 매우 자주 호출
- 근거:
  - cycle/profile 계열에서 다수 `output_data`/`to_excel` 호출 (약 10980~11300행 다수)
- 개선:
  - 메모리 버퍼(DataFrame dict)로 누적 후 시트 단위 일괄 write
  - 필요 시 CSV 우선 + 엑셀 후처리 옵션 분리
- 기대효과:
  - 저장 단계 시간 단축, UI block 완화

## 3) 성능 병목 (난이도 / 개선 기대 퍼포먼스)

| 병목 항목 | 위치(근거) | 난이도 | 기대 성능 개선 |
|---|---|---|---|
| Toyo DCIR 파일 반복 읽기 | 837~847행 | 중 | I/O 30~70% 감소 (재분석 시 체감 큼) |
| `groupby().apply()` 사용 | 813행 | 중 | 전처리 CPU 20~40% 감소 |
| PNE 집계 중복(`pivot_table`) | 1827~1835, 2003행 | 중 | 집계 CPU 15~35% 감소 |
| 루프 내부 잦은 `to_excel` | 10980행대 이후 다수 | 중~상 | 저장 단계 30~60% 감소 |
| UI 셀 반복 재생성(`setItem`) | 13317~13368, 13473~13522행 | 하~중 | 현황 탭 갱신 지연 20~50% 감소 |
| 진행률 업데이트 과빈도 | 10491, 10660, 11918행 등 | 하 | UI 스레드 부하 완화, 끊김 감소 |
| 스캔/경로 탐색 반복 | 10467, 10542, 11900, 12060행 등 | 하 | 대경로 환경에서 초기 지연 완화 |

### 우선순위 제안 (성능 관점)
1. Toyo/PNE 전처리 함수의 I/O + 집계 구조 개선 (`toyo_cycle_data`, `pne_cycle_data`)
2. 엑셀 저장 파이프라인 배치화
3. 현황 탭 테이블 렌더링 최적화(`QTableWidget` -> 모델 기반)
4. 진행률/상태 메시지 체계화

## 4) 그 외 중요도 높은 개선점

### 4-1. 보안/운영 리스크: 네트워크 드라이브 자격증명 하드코딩
- 현황:
  - 계정/비밀번호가 코드에 직접 포함
- 근거:
  - `mount_toyo_button`, `mount_pne1_button`, `mount_pne2_button` (약 13199~13206행)
- 개선:
  - 환경변수/OS Credential Manager/암호화 설정 파일로 분리
  - 실패 시 재시도 및 사용자 피드백 표준화
- 중요도: 매우 높음 (보안)

### 4-2. 경로 하드코딩 정리
- 현황:
  - `d:/`, `D://`, 드라이브 문자 고정 경로 다수
- 근거:
  - 파일 저장/초기 경로/출력 그림 경로 전반
- 개선:
  - 프로젝트 설정(상대경로) + 최근 경로 기억 + pathlib 통일
- 중요도: 높음 (이식성/운영 안정성)

### 4-3. 데이터 처리와 UI 로직 분리
- 현황:
  - 버튼 핸들러 내 전처리/집계/저장/플롯이 결합
- 근거:
  - `indiv/overall/link/continue/dcir` 핸들러 대형 함수 구조
- 개선:
  - `service layer`(I/O, transform, export) 분리
  - UI는 상태 전이와 표시만 담당
- 중요도: 높음 (유지보수/테스트성)

### 4-4. 테스트 부재 구간 보강
- 권장 테스트:
  - Toyo/PNE 사이클 맵핑 일치 테스트
  - DCIR 계산 회귀 테스트(샘플 fixture)
  - profile batch와 단건 결과 동일성 검증
- 중요도: 높음 (회귀 방지)

## 결론
- 단기 체감 개선은 **현황 탭 렌더링 최적화 + 사이클데이터 진행률 가시화 + Toyo DCIR I/O 캐시** 조합이 가장 효과적임.
- 중기 성능 개선의 핵심은 **전처리 벡터화 + 저장 파이프라인 배치화**.
- 운영 안정성 측면에서는 **자격증명/경로 하드코딩 제거**가 최우선 리스크 대응 과제임.
