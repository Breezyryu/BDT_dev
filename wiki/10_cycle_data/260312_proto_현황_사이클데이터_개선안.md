# proto 기준 현황 탭 / 사이클데이터 탭 개선안

> 작성일: 2026-03-12  
> 기준 파일: `DataTool_dev/DataTool_optRCD_proto_.py`

## 목적

`proto` 기준으로 현황 탭과 사이클데이터 탭의 개선 가능 지점을 정리한다.

- UX / UI 개선
- 데이터 처리 속도 개선
- 성능 병목과 난이도 / 기대 퍼포먼스
- 그 외 중요도 높은 구조 개선

## 범위와 전제

- 분석 기준은 `DataTool_dev/DataTool_optRCD_proto_.py` 단일 파일 구현이다.
- 현황 탭은 `tb_summary`, `tableWidget`, `tb_channel`, `FindText` 중심의 화면과 관련 메서드를 기준으로 봤다.
- 사이클데이터 탭은 `cycle_tab`, `indiv/overall/link` 계열 버튼, 병렬 로딩, 그래프 탭 생성 흐름을 기준으로 봤다.
- 아래 개선안은 즉시 적용 가능한 항목과 중장기 리팩토링 항목을 함께 포함한다.

## 핵심 결론

1. 현황 탭은 "데이터 조회 + 검색 + 상태 인지"의 기본 기능은 갖췄지만, 현재는 `QTableWidget` 전체 재구성과 고정 레이아웃 때문에 확장성과 반응성이 낮다.
2. 사이클데이터 탭은 기능은 강하지만 버튼 수가 많고 흐름이 분산되어 있어, 숙련자에게는 강력하지만 신규 사용자에게는 진입 장벽이 높다.
3. 성능은 이미 일부 병렬화되어 있으나, 실제 대기시간의 큰 비중은 여전히 `read_csv/pd.concat/엑셀 저장/그래프 생성`이 차지한다.
4. 가장 큰 구조 리스크는 "UI 이벤트 핸들러 안에 파일 탐색, 데이터 변환, 그래프 생성, 저장 로직이 한꺼번에 섞여 있는 점"이다.

## 1. 현황 탭 개선

### 1-1. UX / UI 개선

| 항목 | 현재 상태 | 개선 방향 | 기대 효과 |
|------|-----------|-----------|-----------|
| 고정 해상도 레이아웃 | `tabWidget`, `tb_channel`, `tableWidget`가 고정 크기 중심으로 구성됨 | `QSplitter`, stretch, resize policy 기반 반응형 레이아웃으로 전환 | 해상도/배율 차이에 덜 민감, 유지보수 용이 |
| 검색 가시성 부족 | `FindText`는 placeholder 설명이 있으나 결과 개수, 활성 조건, 검색 모드 표시가 없음 | 검색 결과 수, 필터 상태, 최근 검색어 표시 추가 | 검색식 학습 비용 감소 |
| 검색 결과 표현 방식 | 비매칭 셀을 흰색 배경과 회색 글자로 처리 | 매칭 개수 배지, "매칭만 보기" 토글, 색상 범례 추가 | 대형 채널판에서 가독성 향상 |
| 요약 정보 부족 | `tb_summary`는 2개 수치만 보여줌 | 사용중/대기/완료/이상/무셀 등 상태별 카운트 확장 | 상태 파악 시간 단축 |
| 상태 컬러 의미 비노출 | 배경색과 글자색 규칙은 있으나 화면에 기준 설명이 없음 | 상단에 색상 legend 추가 | 오해 방지 |
| 새로고침 동선 부족 | 조회 갱신이 콤보박스 재선택 또는 검색 재실행 중심 | 명시적 `새로고침` 버튼, 자동 갱신 주기 옵션 추가 | 운영 화면 성격 강화 |
| 오류 피드백 약함 | 일부 JSON 오류는 콘솔 `print`로만 출력됨 | 사용자용 에러 다이얼로그 + 실패 채널 표시 | 무반응처럼 보이는 상황 방지 |

### 1-2. 구현상 관찰 포인트

- 현황 탭은 `QTableWidget` 셀을 매번 새로 만들고 색상/폰트/테두리를 셀 단위로 다시 설정한다.
- 검색은 실제 필터링이 아니라 "비매칭 셀을 흐리게 만드는 방식"이라, 데이터량이 늘수록 사용자가 필요한 정보만 보기는 어렵다.
- `tb_cycler_combobox()` 호출 시마다 전체 테이블을 다시 채운다.
- `Toyo/PNE`별 색상 규칙과 컬럼 가공 규칙이 메서드 내부에 중복적으로 박혀 있다.

### 1-3. 우선 적용 추천

| 우선순위 | 개선안 | 난이도 | 비고 |
|----------|--------|--------|------|
| 높음 | 현황 데이터 캐시 도입 후 검색/컬럼 변경 시 재파싱 없이 재렌더링 | 중 | 체감 개선이 큼 |
| 높음 | `QTableWidget`에서 `QTableView + QAbstractTableModel`로 전환 | 중~상 | 구조 개선 효과가 큼 |
| 중 | 검색 결과 수, 상태 legend, 새로고침 버튼 추가 | 하 | UX 개선 대비 구현 부담 작음 |
| 중 | 상태 요약 카드 확장 | 하 | 운영 화면 완성도 상승 |

## 2. 사이클데이터 탭 개선

### 2-1. UX / UI 개선

| 항목 | 현재 상태 | 개선 방향 | 기대 효과 |
|------|-----------|-----------|-----------|
| 기능 밀집도 과다 | 좌측 패널에 옵션, 라디오, 체크박스, 버튼이 과도하게 많음 | `사이클`, `프로파일`, `출력` 섹션으로 묶고 단계형 레이아웃으로 재구성 | 진입 장벽 완화 |
| 버튼 의미 불명확 | `indiv`, `overall`, `link`, `link_indiv`, `link_overall`의 차이를 화면만 보고 이해하기 어려움 | 버튼명을 작업 목적 중심으로 변경하고 설명 툴팁 추가 | 잘못된 실행 감소 |
| 탭 제목 정보 부족 | 결과 탭이 `0`, `1`, `2` 같은 번호로 추가됨 | 파일명/조건명/모드명을 포함한 탭 제목 사용 | 다중 결과 비교 시 편의성 상승 |
| 경로 입력 UX 불균일 | 파일 선택, 폴더 선택, 직접 입력(`stepnum_2`)이 혼재 | 입력 소스를 하나의 "데이터셋 선택 패널"로 통합 | 사용 흐름 단순화 |
| 진행률 정보 부족 | 단일 progress bar만 사용, 현재 단계 설명 없음 | `로딩 중 / 병합 중 / 플롯 생성 중 / 저장 중` 상태 텍스트 추가 | 대기 불안 감소 |
| 취소 기능 부재 | 긴 작업 중 중단 버튼 없음 | `취소` 버튼과 작업 토큰 추가 | 장시간 작업 UX 개선 |
| 채널 제어 discoverability 부족 | `▶ CH` 버튼은 기능을 아는 사용자에게만 유효 | `채널 제어` 텍스트 버튼 또는 첫 실행 안내 추가 | 고급 기능 접근성 향상 |
| 결과 재실행 흐름 부족 | 조건 바꿔 다시 실행 시 이전 탭 관리가 수동 | 재실행 시 현재 탭 덮어쓰기/새 탭 생성 옵션 제공 | 탭 난립 방지 |

### 2-2. 구현상 관찰 포인트

- 데이터 로딩은 `_load_all_cycle_data_parallel()`로 병렬화되어 있지만, 그래프 생성과 Excel 출력은 여전히 메인 흐름에 강하게 결합돼 있다.
- `indiv`, `overall`, `link` 계열 메서드는 서로 매우 유사한 패턴을 반복하고 있다.
- figure/canvas/toolbar 생성이 시나리오별로 반복된다.
- 저장 옵션이 켜져 있으면 분석과 동시에 Excel 출력이 끼어들어 총 대기시간이 길어진다.
- 결과 탭 제목이 숫자 기반이라 여러 탭을 만든 뒤 다시 찾기 어렵다.

### 2-3. 우선 적용 추천

| 우선순위 | 개선안 | 난이도 | 비고 |
|----------|--------|--------|------|
| 높음 | `indiv/overall/link` 실행 흐름을 공통 파이프라인으로 통합 | 상 | 중복 제거와 안정성 향상 |
| 높음 | 작업 단계 텍스트 + 취소 기능 추가 | 중 | UX 체감이 큼 |
| 높음 | 탭 제목을 데이터셋명 기반으로 변경 | 하 | 작은 수정으로 큰 편의 |
| 중 | 좌측 옵션 패널을 섹션형 UI로 재배치 | 중 | 신규 사용자 친화적 |
| 중 | 경로 선택 방식을 단일 데이터셋 매니저로 통합 | 중~상 | 장기적으로 중요 |

## 3. 데이터 처리 속도 개선 포인트

### 3-1. 현황 탭

| 개선 포인트 | 현재 문제 | 개선 방향 | 기대 효과 |
|-------------|-----------|-----------|-----------|
| cycler 상태 데이터 캐시 | 콤보 변경/검색마다 JSON 또는 상태 데이터를 다시 가공 | cycler별 DataFrame 캐시, 수정시간 기반 무효화 | 조회 전환 속도 개선 |
| 전체 셀 재구성 제거 | 모든 셀을 새 `QTableWidgetItem`으로 다시 생성 | 모델 기반 렌더링 또는 변경 셀만 갱신 | UI redraw 비용 감소 |
| 문자열/색상 계산 반복 최소화 | 셀마다 동일 규칙 계산 반복 | 상태별 brush/font/color 사전 계산 | CPU 사용량 감소 |
| 검색 디바운스 | 검색어 입력 후 즉시 전체 화면 재계산 가능 | 200~300ms debounce 적용 | 불필요한 연산 감소 |

## 4. 성능 병목 정리

| 병목 | 근거 | 난이도 | 개선 기대 퍼포먼스 | 비고 |
|------|------|--------|--------------------|------|
| 현황 탭 전체 셀 재생성 | 채널판 전체를 셀 단위로 다시 채움 | 중 | 체감 2배 이상 응답성 개선 가능 | 검색/컬럼 전환에 특히 효과적 |
| cycler 상태 데이터 재파싱 | 조회 시마다 JSON/dat 가공 반복 | 중 | 조회 전환 2~5배 개선 가능 | 캐시 적중률이 높을수록 효과 큼 |
| `pd.concat`의 루프 내 반복 | 프로파일/사이클 병합 코드 전반에 다수 존재 | 하 | 해당 구간 1.5~3배 개선 가능 | 가장 먼저 손대기 좋음 |
| Excel 저장이 분석 파이프라인에 포함 | `to_excel`이 작업 시간에 직접 포함 | 중 | 저장 옵션 사용 시 총 시간 30~70% 단축 가능 | UX 효과도 큼 |
| 그래프 생성 메인스레드 집중 | 로딩 후 plot/legend/layout가 한 번에 실행 | 중~상 | 체감 대기시간 20~40% 개선 가능 | 백그라운드 처리와 함께 봐야 함 |
| 재실행 캐시 부재 | 같은 데이터셋도 매번 처음부터 로딩 | 상 | 반복 분석 5~20배 개선 가능 | 분석 툴 특성상 투자 가치 높음 |
| `xlwings` 기반 Excel 읽기 | COM 구동 비용이 큼 | 중 | 특정 시나리오 2~10배 개선 가능 | 승인 사이클 계열에서 효과 큼 |
| 경로 스캔 반복 | `os.scandir/os.listdir`가 여러 흐름에 반복 등장 | 하 | 전체 시간 5~15% 개선 가능 | 단독 효과는 작지만 누적 효과 있음 |

### 가장 먼저 건드릴 병목 3개

1. 루프 내부 `pd.concat` 제거
2. Excel 저장을 분석 단계에서 분리
3. 현황 탭 상태 데이터 캐시 + 모델 기반 렌더링 전환

## 5. 그 외 중요도 높은 개선점

### 5-1. 구조 / 유지보수성

| 항목 | 문제 | 중요도 |
|------|------|--------|
| 단일 대형 파일 구조 | UI, 파일 I/O, 데이터 가공, 시각화가 하나의 파일에 집중 | 매우 높음 |
| 유사 메서드 중복 | `bm_set_profile_button`, `bm_set_cycle_button`이 중복 정의됨 | 매우 높음 |
| 기능별 공통 함수 부족 | `indiv/overall/link` 계열이 유사 코드를 반복 | 높음 |
| 절대경로/드라이브 의존 | `d://`, `D://`, `z:\Working\` 등 하드코딩 다수 | 매우 높음 |
| 콘솔 출력 위주 에러 처리 | 일부 실패가 UI에 드러나지 않음 | 높음 |

### 5-2. 왜 중요한가

- 중복 메서드는 한쪽만 수정되는 회귀 버그를 만들기 쉽다.
- 절대경로 하드코딩은 다른 PC, 다른 드라이브 구성, 배포 EXE 환경에서 장애를 만든다.
- 현재 구조는 새 기능 추가보다 "기존 기능을 건드릴 때 깨질 위험"이 더 큰 단계에 가깝다.

## 6. 추천 실행 순서

### Phase 1. 빠른 개선

- 현황 탭 검색 결과 수, 상태 legend, 새로고침 버튼 추가
- 사이클데이터 탭 탭 제목 개선
- `pd.concat` 반복 구간 list 누적 방식으로 변경
- Excel 저장 단계 분리

### Phase 2. 체감 성능 개선

- 현황 탭 cycler 데이터 캐시 도입
- 진행률 텍스트, 취소 기능 추가
- 반복되는 cycle 실행 로직 공통화
- 경로 선택 UI를 데이터셋 단위로 정리

## 최종 우선순위 제안

가장 투자 대비 효과가 큰 항목은 아래 5개다.

1. 현황 탭 데이터 캐시 + 전체 재렌더링 축소
2. 사이클데이터 탭 저장 단계 분리
3. `indiv/overall/link` 공통 파이프라인 정리
4. 루프 내부 `pd.concat` 제거
5. 절대경로/중복 메서드 정리

이 5개만 정리해도, 사용자 체감은 "검색과 조회가 빨라지고, 긴 작업이 덜 답답하며, 유지보수 위험이 낮아진다"는 방향으로 분명하게 바뀔 가능성이 높다.
