---

## 1. 배경 및 목적

DataTool은 PNE 장비 데이터를 CSV 파일(`SaveEndData.csv`, `SaveData000N.csv`)로 불러와 사이클 분석을 수행한다. 

PNE 장비는 CSV 내보내기와 별도로 바이너리 포맷 `.cyc` 파일을 실시간으로 기록한다. `.cyc`의 핵심 장점은 **CSV보다 약 1일 최신 데이터**를 포함한다는 것이다. 이 분석은 `.cyc`를 직접 파싱하여 기존 CSV 기반 워크플로를 대체할 수 있는지 검토한다.

## 3. .cyc 구조적 특성

### 3.1 스텝 경계 감지

`.cyc`에서 스텝 경계를 감지하는 방법: **`fid6 (StepTime) == 0.0` 인 레코드 = 새 스텝 시작**

```
rec  82: fid6=4840.65  ← discharge 스텝 마지막
rec  83: fid6=0.00     ← rest 스텝 시작 (경계!)
rec  84: fid6=0.10
...
rec  94: fid6=600.00   ← rest 스텝 마지막
rec  95: fid6=5440.65  ← 루프 마커 레코드 (특수)
rec  96: fid6=0.00     ← charge 스텝 시작 (경계!)
```

### 3.2 루프 마커 레코드 (StepType=8)

루프 마커는 `.cyc`에 **단일 레코드**로 저장되며, 다음 두 가지 감지 방식이 필요:

| 감지 방법 | 조건 | 사이클 유형 |
|----------|------|------------|
| 방법 A: fid6 점프 | `fid6[-1] - fid6[-2] > 300 sec` | 긴 사이클 (루프 누적 시간 > 스텝 시간) |
| 방법 B: DchgCap 비교 | 주 구간 DchgCap=0 AND 마지막 레코드 DchgCap > 100 mAh | 짧은 사이클 (루프 누적 시간 < 스텝 시간) |

> **주의:** 방법 A만 사용하면 짧은 체크포인트 사이클에서 루프 마커 감지 실패 → TotlCycle 오류 발생.

### 3.3 StepType 추론 규칙

```
평균 전류 > +10 mA  → 1 (충전)
평균 전류 < -10 mA  → 2 (방전)
|평균 전류| ≤ 10 mA → 3 (휴지) 또는 8 (루프 마커, 별도 감지)
```

### 3.4 EndState 추론 규칙

| 조건 | 추론 EndState | 근거 |
|------|------------|------|
| StepType=3 또는 8 | 64 (휴지) | 항상 정확 |
| StepType=2 (방전) | 65 (전압종료) | 표준 사이클에서 ES=78 미발생 |
| StepType=1 (충전), \|EndCurr\| < C/45 | 66 (전류종료) | CC/CV 완료 |
| StepType=1 (충전), \|EndCurr\| ≥ C/45 | 65 (전압종료) | CC 단계 전압 도달 |

## 4. 검증 결과 (전체 6,789행 비교)

대상: TotlCycle=1~759 (CSV 완성 사이클 전체)

| 항목 | 정확도 | 비고 |
|------|--------|------|
| **TotlCycle** | **100.00%** | 사이클 번호 완전 일치 |
| **StepType** | **100.00%** | 충전/방전/휴지/루프 완전 일치 |
| **EndState** | 77.93% | ES=65↔66 불일치 1,498건 (DataTool 분류 영향 없음) |
| **DchgCap** | **100.00%** | float32 정밀도 내 완전 일치 |

### 4.1 DataTool 사이클 분류 정확도

`classify_pne_cycles()` 함수는 EndState에서 **`has_es78` (ES=78 유무)** 만 검사한다.

- 이 데이터셋에서 ES=78 발생: **0건 (CSV 6,792행 모두)**
- CYC 추론 ES=78 오탐: **0건**
- **사이클 카테고리 분류 정확도: 100%**

### 4.2 ES=65 vs ES=66 불일치 패턴

불일치 1,498건 모두 동일 패턴:
- `StepType=1 (충전), 실제=66 (전류종료), 추론=65 (전압종료)`
- 이는 멀티스텝 가속충전 스텝에서 PNE가 ES=66으로 코딩하지만, 실제 EndCurr > C/45인 경우
- DataTool 분류 로직에 영향 없음 (사이클 카테고리 판별은 StepType 기반)

## 5. .cyc 파싱 도입 장단점

### 5.1 장점

| # | 장점 | 효과 |
|---|------|------|
| 1 | **최신 데이터** | CSV보다 약 1일 추가 사이클 포함 (본 데이터셋: +46행, TotlCycle=760~765 추가 캡처) |
| 2 | **단일 파일** | `.cyc` 1개로 모든 스텝 데이터 포함 (CSV는 93개 파일) |
| 3 | **실시간 파싱 가능** | 실험 진행 중 파일 잠금 없이 읽기 가능 |
| 4 | **용량 효율** | 93개 CSV 합산 대비 작은 파일 수 |
| 5 | **binary 포맷 일관성** | 엑셀 특수문자 등 CSV 파싱 오류 없음 |

### 5.2 주의사항 및 한계

| # | 항목 | 상세 |
|---|------|------|
| 1 | **EndState 불완전 추론** | ES=65 vs ES=66 구분 ~78% 정확도. 추후 ES=78 (Rss) 필요 시 별도 로직 필요 |
| 2 | **ChgCap 없음** | .cyc에 충전용량 필드 미기록. 현재 DataTool 코드에서는 미사용이므로 영향 없음 |
| 3 | **float32 정밀도** | 전압 ±1 μV 오차 가능. 단, 정수 비교가 아닌 ≤0.1% 오차 허용 시 100% 일치 |
| 4 | **루프 마커 감지 복잡성** | 체크포인트(특수 짧은 사이클) 대응 위해 DchgCap 기반 감지 필수 |
| 5 | **단위 변환 필수** | .cyc는 mV/mA/mAh/°C, CSV는 μV/μA/μAh/m°C (×1000 변환) |
| 6 | **Rss 사이클 감지 불가** | ES=78을 추론하지 않으므로 `has_es78=True` Rss 사이클 분류 불가 (현 알고리즘) |

## 6. SaveEndData 재구성 알고리즘 요약

구현 파일: `analysis_dev_env/cyc_parser.py`

```
1. .cyc 헤더 파싱 → FieldID, data_start, rec_size
2. 전체 레코드 numpy float32 배열 로딩
3. fid6 == 0.0 레코드 위치 → 스텝 경계 목록
4. 각 스텝 구간 처리:
   a. 루프 마커 감지 (방법 A + 방법 B)
   b. 메인 스텝 마지막 레코드에서 값 추출
   c. StepType 추론 (전류 방향)
   d. EndState 추론 (전류 크기 기준)
   e. TotlCycle 증가 (루프 마커 발생 시)
5. DataFrame 반환 (DataTool summary 포맷과 동일한 컬럼 구조)
```

## 8. 결론

`.cyc` 파싱으로 SaveEndData 재구성은 **DataTool 핵심 분석 기능(사이클 카테고리 분류) 기준으로 100% 정확도**를 달성한다. `.cyc`는 CSV보다 최신 데이터를 포함하고 파일 관리도 단순하여, 기능 추가 시 **운영 효율 향상** 효과가 명확하다.

주요 미해결 과제는 ES=65 vs ES=66 구분 (분류에 영향 없음)과 ES=78 (Rss) 추론이다. Rss 테스트가 포함된 데이터에 대해서는 추가 개발이 필요하다.
