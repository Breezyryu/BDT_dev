1. 개요 — 두 위젯의 역할과 단위 일원화

BDT 사이클 데이터 탭의 운영 체계는 두 핵심 위젯의 단위 정합성으로 정의된다:

① 사이클 타임라인 바 L11026 CycleTimelineBar

  • 경로별 충방전 패턴을 색상 블록으로 시각화
  • 행별 독립 선택 — 다중 경로 시 행마다 별도 선택 유지
  • 클릭/Shift/Ctrl/우클릭 메뉴 + 눈금 + 툴팁
  • 입력 단위 = TC (TotlCycle 의미상 등가)

② 경로 테이블 cycle_path_table 7 cols

  • 경로 입력 + 메타 자동 채우기 + TC 범위 입력
  • 연결처리 토글로 그룹·누적 운영
  • 빈 행이 다중 그룹 구분자 역할
  • 입력 단위 = TC (col4), 누적 단위는 그룹별 TC 합

두 위젯은 같은 TC 단위로 동기화된다. 사용자가 col4에 입력한 값은 타임라인 바 선택을 갱신하고, 바 클릭은 col4를 갱신한다 (_timeline_syncing 플래그로 이중 갱신 차단).

사이클 번호 통일 기준 (260412 정비) — 사용자 레이어 = 논리사이클(=TC), 내부 데이터 매핑 레이어 = TC. CycleNo single source = col4.

3. 사이클 타임라인 바 — 위젯 구조

3.1 행 모델 — 단일경로 vs 다중경로

단일경로 (1 row)

set_blocks(blocks)
→ self._rows = [('', blocks, total)]
→ _active_row = 0
높이: 18px (1 row)

다중경로 (N rows, 최대 4행 가시)

set_multi_blocks([(label, blocks), ...])
→ self._rows = [(label, blocks, total), ...]
→ _row_selections: dict[int, list[(s,e)]]
높이: 자동 (N×18 + N-1×2 + 16 tick + 4)
스크롤: _max_h = 4-row 한계 초과 시

활성 행 전환 — 다른 행 클릭 시 _active_row 변경, 이전 선택은 유지(파란 오버레이로 표시), 새 활성 행 선택은 흰색+노란 테두리.

3.2 블록 빌드 파이프라인

_update_cycle_timeline(self, raw_file_path=None) ├─ pne_path_setting() → folders 목록 ├─ for folder in folders: │ ├─ scandir → 채널 폴더 정렬 │ ├─ Phase 0 메타 기반 max_tc 가장 큰 채널을 대표 채널로 선정 │ ├─ blocks 생성: │ │ ├─ classified 있음 → _build_timeline_blocks_tc_by_loop ← 우선 │ │ ├─ PNE classified 없음 → _build_timeline_blocks(SaveEndData) ← 폴백 │ │ └─ Toyo classified 없음 → _build_timeline_blocks_toyo(channel_path) │ └─ rows.append((label, blocks)) ├─ link_mode + len(rows) > 1: │ └─ _get_table_row_groups() → 그룹별 블록 누적 합산 + offset 적용 └─ rows 1개 → set_blocks / 다수 → set_multi_blocks

3.4 시각 표현 규약

  • 활성 행: 진한 회색 테두리 + 선택 영역 흰색 반투명 + 노란 테두리
  • 비활성 행: 옅은 회색 테두리 + 선택 영역 파란 반투명 + 파란 테두리
  • 반사이클 패턴 (initial, 반사이클): 시각 가중치 0.3배 — 짧게 표시
  • 최소 블록 폭: 8px — 짧은 블록도 가독성 보장 (긴 블록에서 비례 분배)
  • C·E 패턴, count ≤ 2: 색칠 대신 가운데 흰색 세로선 (시작/종료 마커)
  • 눈금 step: tc≤20→5 / ≤100→10 / ≤500→50 / ≤2000→100 / >2000→500

4.2 자동 채우기 모드 — Light vs Full

LIGHT IO 0, regex만

  • 대상 컬럼: 0 시험명 + 3 용량
  • 호출 시점: 경로 입력 직후 (paste / drop / cellChanged / load)
  • 비용: 0 ms (basename + name_capacity 정규식)
  • 캐시: _path_meta_cache_light (인스턴스 dict)

FULL scandir + cycle_map

  • 대상 컬럼: 02346 전부
  • 호출 시점: "사이클 분석" 버튼 클릭 (confirm 진입부)
  • 비용: 100 ms ~ 5 s/경로 (cycle_map 빌드 포함)
  • 캐시: _path_meta_cache + lru_cache(check_cycler/_quick_max_cycle)
  • UI 응답: statusBar "채널·TC 채우기 중... (k/n)" + processEvents()

5. 운영 케이스 6종 — 단일·다중·연결·다중그룹·ECT·첫 분석 가드

경로 테이블의 입력 패턴 + 분석 시점 가드 동작에 따라 6가지 케이스가 발생한다. A~E는 정적 입력 모드, F는 분석 시점에 발생하는 동적 UX 케이스(=v2 guard 상호작용).

케이스차원핵심 특징
A 단일정적 (입력)1행, 사이클바 1행
B 다중 OFF정적 (입력)N행, 사이클바 N행, 각 행 독립
C 연결 단일그룹정적 (입력)N행 합산, 사이클바 1행 (offset)
D 연결 다중그룹정적 (입력)빈 행 구분, 그룹 수 = 사이클바 행
E ECT정적 (입력)chk_link_cycle 강제 OFF, col5 활성
F 첫 분석 가드동적 (분석 시점)v2 guard 자동 col4 덮어쓰기 → user input 유실 vector
